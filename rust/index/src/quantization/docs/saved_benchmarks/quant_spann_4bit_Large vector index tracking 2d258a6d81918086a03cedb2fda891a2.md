This note presents some basic estimation on key metrics of large vector index.

Baseline collection, 1M 4096D F32:

- 2^20 vectors, 2^12 dimensions, 2^2B per dimension
- Embedding size: 2^34B = 16GB
- Existing SPANN implementation:
    - ~2^2 copies per vector on average
    - writer amplification (keep existing & new block), assume to be a factor of ~2
    - ~2^37B = 128GB memory requirement during compaction
    - ~2^6 vecter per cluster, HNSW index needs to store 2^16 centroids, which means query needs to load 2^30B = 1GB before query evaluation can start

Minimal scalability requirement, 64M 4096D F32:

- Embedding size: 2^40 bytes = 1TB
- Existing SPANN implementation:
    - need 2^43B = 8TB space during compaction
    - needs to pull 64GB before query can start
- If we need to compact in 8 hours, we need throughput of 8M per hour or 2M per 15min

Moderate scalability requirement, 1B 4096D F32:

- Embedding size: 2^44B = 16TB

Ideal scalability requirement, 128B 4096D F32:

- Embedding size: 2^51B bytes = 2PB

Available techniques in toolbox:

- RabitQ: 1Bit quantization reduce memory and storage by 2^5
- Existing SPANN implementation specific:
    - Normalization: keep one copy per vector, reduce memory and storage by ~2^2
    - More aggressive clustering: more vector per cluster, say 2^8 = 256 vector per cluster. Reduces HNSW index size by 2^2
    - In the best scenario, for the minimal scalability requirement:
        - We resolved the slow compaction time, this needs to be further investigated and we cannot know how good we can do in advance
        - Use RabitQ + Normalization + Larger cluster
            - Compaction space requirement: 2^36B = 64GB, can fit in memory
            - Query HNSW size: 2^29B = 512MB, acceptable for cold query

=== 1/21/2026 ===

Current setup: ~50 split threshold, ~25 merge threshold, read nprobe 64, support up to ~15M without issue already

Goal: support up to ~50M asap

- No major overhaul to existing impl

Main bottlenecks:

- Centroid index is getting large, query hnsw cache seems to be thrashing now, cold latency is high. If we scale up to ~50M the centroid index will be 3x larger and the issue will be worse
- Compaction speed is a bit slow. We need a long time to finish compaction on 50M
- Compaction memory usage might be high. At 50M we have 50M vecs * 4K dim/vec * 4B/dim = 800GB data. with ~5x replication factor the memory usage might not be tolerable if we need to load everything in memory.

Assumption: single blockfile is robust enough to scale to this data size

Proposed quick solution:

- To reduce centroid index size, the options include:
    - Reduce size per vector, via quantization
    - Reduce number of centroids, with larger postings and smaller replica count
- To speed up compaction, the options include:
    - Reduce replica count, which empirically speeds up compaction
    - Increase posting list size, which empirically speeds up compaction
    - Introduce append() in blockfile
    - Reduce Trim()
    - Pruning Neighbor search during reassign
    - Replace HNSW with USearch, which empirically increases level of concurrency
- To reduce memory usage, the options include:
    - Normalization: deduplicate vectors loaded from blockfiles, and blockfile must not pin the posting lists in memory after read
    - Delta encoding in index: we do not eagerly load vectors to index, but only when we need to perform Scrub operation. we only keep deltas in memory and flush them in the end. (to be prototyped).
- Correlated:
    - Byte range read from blockfile, unblocks larger block size and thus larger posting list size

Personal opinion @Sicheng Pan :

From most cost-effective to least:

- Increase cluster size a bit, reduce replica count
    - Each block in blockfile needs to hold at least 2 posting lists. we need to pull the entire block for query.
- Quantization
    - On centroid, yields 8x size reduction
    - On cluster, yields 8x size reduction, and unlock larger posting lists
- Normalization in memory (on disk format stay the same)
    - Reduced memory usage, so that larger collection can fit in memory
